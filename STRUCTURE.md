## 🗂️ Directory Breakdown

### **`/builds`** — Compiled CloudStream 3 Provider Plugins
- **What**: Pre-built `.cs3` provider plugin files (compiled from decompiled sources)
- **Why tracked**: These are the production artifacts deployed via `repo.json`
- **Contents**: 19 `.cs3` files (one per provider)
- **Example**: `builds/Xhamster.cs3`, `builds/iGay69.cs3`
- **Generated by**: Kotlin compilation targeting CloudStream 3.7+ API

---

### **`/.github/workflows`** — GitHub Actions CI/CD
- **What**: Automated build and validation pipeline
- **Why tracked**: Needed for continuous integration when code is pushed
- **Files**:
  - `build-cs3.yml` — Main workflow:
    - Compiles all plugins
    - Runs official Cloudstream unit tests
    - Runs fallback provider_e2e_validator.py
    - Quality gate: **fails if ANY provider verdict is FAIL**
    - Uploads artifacts (builds, verification reports)

---

### **`/analysis/scripts`** — Provider Validation Tooling
- **What**: Python scripts that analyze and score each provider
- **Why tracked**: Core validation logic and orchestration
- **Key files**:
  - `provider_e2e_validator.py` — Scoring engine:
    - Detects MainAPI implementations
    - Checks method signals (search, load, loadLinks)
    - Analyzes patterns, selectors, DNS resolution
    - Produces weighted verdict (PASS_STRONG / PASS / FAIL)
    - Outputs: `docs/verification/provider_e2e_validation.json`
  - `run_e2e_validation.sh` — Orchestrator:
    - Runs official tests first (if available)
    - Falls back to `provider_e2e_validator.py`
    - Prints summary to console
  - `launch_dashboard.sh` — Dashboard HTTP server:
    - Serves `docs/dashboard/` on port 8787 (configurable)
    - Provides real-time access to validation results
  - `test_suite.sh` — Test discovery and execution
  - Other utilities: maxlevel_engine.py, morphe_pme.py, etc.

---

### **`/docs/dashboard`** — Interactive Validation UI
- **What**: Single-page web app for visualizing provider health
- **Why tracked**: Provides real-time visualization of validation results
- **Files**:
  - `index.html` — Structure (cards, tables, filters)
  - `styles.css` — Dark theme styling
  - `app.js` — Fetches JSON, renders data, filters by verdict/score
- **How to access**:
  ```bash
  ./analysis/scripts/launch_dashboard.sh
  # Then open: http://127.0.0.1:8787/dashboard/index.html
  ```
- **Data source**: Reads from `docs/verification/provider_e2e_validation.json` (auto-generated)

---

### **`/src`** — Source Code and Templates
- **What**: Kotlin templates and Python tooling source
- **Why tracked**: Foundation for provider generation and tools
- **Subdirectories**:
  - `src/providers/_templates/` — BaseProvider.kt (CloudStream API template)
  - `src/reaver/` — Reaver tool source (pattern extraction, validation)

---

### **`/scripts`** — Build and Utility Scripts
- **What**: Python automation for builds and validation
- **Why tracked**: Build orchestration and tooling
- **Files**:
  - `build_all.py` — Compile all providers to `.cs3`
  - `reaver_browserless.py` — Integration with browserless extraction
  - `reaver_validate.py` — Validate provider structure

---

## 📦 Root Files

### Configuration and Index Files

- **`repo.json`** — CloudStream extension repository manifest
  - Format: JSON with provider metadata
  - Used by: CloudStream app to discover/download providers
  - Updated: After compilation of new providers
  
- **`plugins.json`** — Provider index/reference
  - Format: JSON array of provider names
  - Purpose: Quick lookup of available providers

- **`README.md`** — Repository documentation
  - Usage: Getting started, validation plan, dashboard access
  - Audience: Developers cloning the repo

- **`STRUCTURE.md`** (this file) — Directory guide
  - Purpose: Explain what everything does and why

- **`.gitignore`** — Git ignore patterns
  - Excludes generated data, dependencies, logs
  - Prevents large/ephemeral files from being tracked

### Executable Scripts

- **`reaver.sh`** — Main entry point for Reaver tool
  - Purpose: Pattern extraction and plugin validation
  - Usage: `./reaver.sh [command] [args]`

- **`reaver_patcher.py`** — Plugin patching utility
  - Purpose: Apply fixes to extracted providers
  - Generates: `reaver_patches/` (gitignored)

- **`telemetry.sh`** — Monitoring and logging
  - Purpose: Track validation runs and health metrics
  - Outputs: Local logs (gitignored)

## 📊 Gitignored Directories (Generated/Large Data)

These directories are **NOT tracked** but are essential for development and testing:

### **`/data`** — Extracted Provider Analysis Data
- Contains: Decompiled source, patterns, selectors, strings per provider
- Size: ~15 MB
- Generated by: Chimera extraction pipeline
- Purpose: Reference material for provider analysis
- Example structure:
  ```
  data/chimera_extracted_v3/
  ├── Xhamster_jsm.txt          # JavaScript/method signatures
  ├── Xhamster_patterns.txt      # URL patterns
  ├── Xhamster_selectors.txt     # CSS/XPath selectors
  ├── Xhamster_jadx/sources/     # Decompiled Kotlin/Java
  └── ...
  ```

### **`/browserless`** — Browserless.io Distribution
- Contains: Full browserless/... npm package
- Size: ~148 MB
- Purpose: Browser automation for provider testing/extraction
- Generated by: `npm install` or manual setup
- Not tracked: Because it's large and can be regenerated

### **`/upstream`** — CloudStream Reference Repository
- Contains: Clone of official Cloudstream 3 repo for testing/reference
- Size: ~594 MB
- Purpose: Official test harness, API reference
- Generated by: Manual clone from GitHub
- Not tracked: Because it's a full external repository clone

### **`/reaver_patches`** — Generated Patch Artifacts
- Contains: Patch files from `reaver_patcher.py`
- Purpose: Version control for incremental provider fixes
- Not tracked: Because these are working artifacts

### **`/docs/reports`** and **`/docs/verification`** (partial)
- Generated outputs from validation runs
- Not tracked: Because they change frequently
- Manually reviewed but not committed as source

## 🔄 Workflow: How Everything Connects

### **Development Cycle**

```
1. Edit provider source or templates (src/)
   ↓
2. Run build: python3 scripts/build_all.py
   ↓
3. Compile Kotlin: builds/*.cs3 are created
   ↓
4. Validate: python3 analysis/scripts/provider_e2e_validator.py
   ↓
5. Review results: docs/verification/provider_e2e_validation.json
   ↓
6. View dashboard: ./analysis/scripts/launch_dashboard.sh
   ↓
7. Commit changes to src/ (git add src/ docs/dashboard/ ...)
```

### **CI/CD Trigger**

```
1. Push to origin/main
   ↓
2. GitHub Actions runs: .github/workflows/build-cs3.yml
   ↓
3. Workflow:
   - Sets up Java 17 + Gradle
   - Compiles all plugins
   - Runs ./gradlew :app:testStableDebugUnitTest (official tests)
   - Runs fallback validator (if official blocked)
   - quality gate: FAILS pipeline if any FAIL verdict
   - Uploads artifacts
```

## 📝 Notes

1. **Data directories are local**: `data/`, `browserless/`, `upstream/`, `reaver_patches/` exist locally but won't be committed. They can be regenerated as needed.

2. **Tracked files are minimal and intentional**: Only source code, configuration, and dashboards are tracked. This keeps the repo size small and efficient.

3. **CI/CD is automated**: Changes to `src/` or analysis scripts trigger automatic validation on push.

4. **Validation is deterministic**: Same provider code produces same validation results (except DNS, which is environment-dependent).

5. **Dashboard is static HTML**: Can be served by any HTTP server; no backend required.
